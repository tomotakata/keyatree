1

Supabase に関するご質問

Q1

データ削除時・更新時の履歴は追えますか。

はい、対応可能です。 標準的なデータベースだけでは「誰が・いつ・何を変更したか」を完全には残しませんが、 今回の開発では、監査ログ用テーブルを追加することを想定しておりますので、 実装を進めることで対応可能です。

具体的には、社員情報・評価・タスクなどの主要テーブルに対して、 更新前データ / 更新後データ / 操作者 / 操作日時 / 操作種別（作成・更新・削除）を別テーブルへ自動記録します。

  • 主要テーブルごとに履歴保存対象を定義可能です。
  • 削除済みデータの確認や、差分比較にも応用できます。
  • 必要に応じて「元に戻す」機能の設計も可能です。
Q2

利用者が約60人となりますが、動作が重くなる等の心配は不要でしょうか。

60人規模のご利用、および一定の同時アクセスが発生する前提でも、適切な設計を行えば十分に安定運用が可能です。 Vercel と Supabase の構成は、もともと同時アクセスに対応しやすいクラウド基盤であり、社員管理・人事評価・タスク管理といった業務システム用途であれば、60名規模は無理のない範囲です。

そのため、「同時に利用者がアクセスした場合に耐えられない」という前提ではなく、 同時アクセスがあっても問題のない構成・実装を前提に設計する方針です。 特に本システムは、動画配信や大規模SNSのような超高負荷サービスではなく、業務データの閲覧・更新が中心となるため、 適切なデータ取得設計と画面最適化を行うことで、60名規模の利用には十分対応可能です。

  • 社員一覧はページネーションや検索条件の最適化により、同時利用時も表示負荷を抑えられます。
  • 集計処理はキャッシュ化・事前計算により、ピーク時の負荷集中を回避できます。
  • 画像や添付ファイルは CDN 配信によりアプリ本体とは分離して高速配信が可能です。
  • 必要に応じて上位プランや監視追加により、将来的な利用増加にも段階的に対応できます。
Q3

将来、他のクラウドサービスへ DB の移行は可能でしょうか。

はい、移行可能です。 Supabase は PostgreSQL ベースのため、 将来的に AWS RDS、Google Cloud SQL、Azure Database for PostgreSQL、 あるいは自社管理の PostgreSQL 環境へ移行しやすい構成です。

ベンダーロックインを最小化するため、アプリケーション設計においても PostgreSQL 標準機能を中心に利用する方針が望ましいと考えております。

  • DB本体は SQL ダンプや移行ツールにより移設可能です。
  • 認証・ストレージ機能は別途移行設計が必要です。
  • 将来の拡張性・可搬性を意識した構成にすることが可能です。

システム運用・保守・ランニング費用について

Q4

障害発生時のサポート体制（連絡手段）、対応時間＝SLA はどのように考えていますか。

想定運用案としては、一次連絡手段は貴社指定のチームメッセージを基本とし、緊急時には電話でのご連絡を想定しております。

基本受付時間は平日 9:00〜18:00としておりますが、現状の開発段階においても深夜帯のご連絡へ対応しており、 今後も状況に応じて継続してご対応予定でございます。

目安としての概算ではございますが、重大障害 2時間以内、中障害 1営業日以内、軽微不具合 3営業日以内を一つの対応目安としております。

復旧対応につきましては、原因切り分け → 暫定対処 → 恒久対策 の順で進める方針です。

保守契約をご検討いただくに際しましては、システムに関する正式な開発項目の確定時に、 正式な SLA として「障害レベル定義」および「対応時間帯」を明文化し、ドキュメント化することを推奨しております。

Q5

リリース後の保守費用と保守範囲を教えてください。

保守費用の考え方としては、上記のような軽微修正、エラー調査、アカウント管理補助、 ライブラリアップデート、バックアップ確認、監視確認などを含め、 ご契約を継続いただく場合には、契約範囲内でインクルードして対応する想定です。

一方で、対応内容が契約範囲を超える場合や、別途対応項目ごとのご相談が必要な場合には、 対象項目ごとに個別お見積りを作成し、ご相談のうえ進める形となります。

参考までに予算の概算としましては、 小規模運用では 月額 3万円〜10万円程度が一つの目安です。 初期フェーズでは「軽保守（障害時対応中心）」、本格運用後は「月次改善・定例対応込み」の 運用保守へ切り替える形が現実的です。

なお、保守対象項目を限定した契約形態の場合には、 大規模改修や新機能追加は保守費用には含まれず、別見積りでのご相談となります。

  • 軽微な不具合修正
  • エラー発生時の原因調査
  • アカウント・権限管理の補助
  • ライブラリ更新・セキュリティアップデート対応
  • バックアップ状況や監視状況の確認
Q6

毎月のランニングコストが増加する条件を教えてください。

前提として、今回開発するシステムにつきましては、利用するユーザーごとに費用が課金されるような形ではなく、自社開発システムのためユーザー単位のライセンス課金はありません。

あくまでも、ご案内しているインフラサーバーの基本利用料のほかに、変動費として想定される費用の対象について、以下に記載しております。

  • 写真・添付ファイルなどの保存容量増加
  • 月間閲覧数・API呼び出し回数の増加
  • 通知メール送信数の増加
  • バックアップ保持期間の延長
  • ログ保存期間の長期化
  • リアルタイム同期や分析集計処理の追加
  • 本番／検証／開発など複数環境の運用
  • 外部サービス連携（LINE、会計ソフト、勤怠システム等）の追加
  • 監視や分析基盤の追加導入
Q7

セキュリティに関するご質問

Q8

IP制限は可能ですか。

はい、可能です。 ただし、ここでのIP制限は、開発するシステムのログイン時におけるアクセス制御と、 インフラ・ネットワーク側でのアクセス制御を分けて考えるのが適切です。

システム側の考え方としては、ログインを試みた際に許可されたIPアドレスからのみ認証処理を受け付ける、 という形での制御が可能です。 たとえば、管理者アカウントのみ社内IPに限定し、一般社員は通常ログインを許可する、といった運用設計も可能です。

一方で、インフラ側の観点では、Vercel、WAF、Reverse Proxy 等で特定IPのみアクセスを許可する構成も選択可能です。 こちらはシステム全体や特定管理画面への到達自体を制御する考え方になります。

なお、外出先やスマートフォンからの利用を想定する場合は、IP制限を厳格にしすぎると利便性が下がるため、 「ログイン制限をかける対象ユーザー」と「インフラ側で制限する対象画面」を切り分けて設計する方針が適切です。

Q9

不正アクセス検知はありますか。

標準機能のみで高度な SOC レベル監視を行うわけではありませんが、 ログイン失敗回数の記録、異常IPアクセスの監視、急激なアクセス増加の検知 など、 不正アクセス兆候を把握する仕組みは構築可能です。

  • 連続ログイン失敗の検知
  • 特定IPからの過剰アクセス検知
  • APIエラー急増の通知
  • Sentry / Logtail / Datadog 等との連携による監視強化
Q10

各ユーザーの操作ログは全て取得可能ですか。

はい、取得可能です。 ログイン、閲覧、作成、更新、削除、承認、通知送信等の各操作を 監査ログテーブルへ記録する設計にすることで、各ユーザーの操作履歴を追跡できます。

  • ユーザーID
  • 対象データID
  • 操作内容
  • 実行日時
  • IPアドレス
  • 端末・ブラウザ情報
  • 変更前後データ（必要に応じて）

なお、保存対象が増えるほど保管容量も増えるため、監査要件と個人情報保護の観点を踏まえ、 保存範囲を要件定義時に整理することが望ましいです。

Q11

バックアップの頻度・保存期間・復旧時間・復旧方法を教えてください。

想定運用としては、日次バックアップを基本とし、 保存期間は 7日〜30日程度を一つの基準と考えております。 要件に応じて、より長期の保存や別クラウドへの二重保存も対応可能です。

  • 頻度：日次（必要に応じて複数回/日も検討可）
  • 保存期間：7日〜30日を想定
  • 復旧時間目安：軽微障害は数十分〜数時間、重大障害は半日〜1営業日
  • 復旧方法：Supabase バックアップ復元、または SQL / CSV エクスポートから再投入

可用性を高める場合は、定期エクスポートを別ストレージへ保管する二重バックアップ体制も構築可能です。
