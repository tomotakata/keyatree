1

Supabase に関するご質問

Q1

データ削除時・更新時の履歴は追えますか。

はい、対応可能です。 標準的なデータベース構成だけでは、更新・削除の履歴をそのまま自動で完全保存するわけではありませんが、 監査ログ（Audit Log）用のテーブルを別途設計することで、 「誰が」「いつ」「何を」「どのように変更したか」を記録することができます。

具体的には、社員情報、評価情報、タスク情報などの主要データに対して、 更新前データ・更新後データ・操作者・操作日時・操作区分（作成／更新／削除）を保存する構成を想定しております。

  • 主要テーブルごとに履歴保存対象を定義可能です。
  • 削除済みデータの確認や、差分比較にも応用できます。
  • 必要に応じて「元に戻す」機能の設計も可能です。
Q2

利用者が約60人となりますが、動作が重くなる等の心配は不要でしょうか。

60人規模であれば、一般的な社員管理・人事評価・タスク管理用途では大きな懸念はございません。 Vercel と Supabase の標準構成でも十分に対応可能な規模です。

想定されるアクセスは、全員が常時高負荷で利用するようなシステムではなく、 業務時間帯に分散して閲覧・更新が発生する運用と考えられるため、 初期構成としては過不足のない選定です。

  • 社員一覧はページネーション化し、表示負荷を抑える設計が可能です。
  • 集計処理はキャッシュ化・事前計算により高速化できます。
  • 画像や添付ファイルも CDN 配信により負荷分散できます。
Q3

将来、他のクラウドサービスへ DB の移行は可能でしょうか。

はい、移行可能です。 Supabase は PostgreSQL ベースのため、 将来的に AWS RDS、Google Cloud SQL、Azure Database for PostgreSQL、 あるいは自社管理の PostgreSQL 環境へ移行しやすい構成です。

ベンダーロックインを最小化するため、アプリケーション設計においても PostgreSQL 標準機能を中心に利用する方針が望ましいと考えております。

  • DB本体は SQL ダンプや移行ツールにより移設可能です。
  • 認証・ストレージ機能は別途移行設計が必要です。
  • 将来の拡張性・可搬性を意識した構成にすることが可能です。
2

システム運用・保守・ランニング費用について

Q4

障害発生時のサポート体制（連絡手段）、対応時間＝SLA はどのように考えていますか。

現時点では、以下のような 実務的な運用体制案 を想定しております。 正式な SLA については、本番導入前に「障害区分」「受付時間」「一次回答時間」「復旧目標時間」を定義し、 契約条件として明文化する形が望ましいと考えております。

  • 連絡手段：メール、Chatwork、Slack、必要に応じて電話
  • 受付時間：平日 9:00〜18:00
  • 重大障害：一次回答 2時間以内を目安
  • 中程度障害：1営業日以内を目安
  • 軽微不具合：3営業日以内を目安

なお、実際の復旧時間は障害内容に依存するため、 「一次回答」と「完全復旧」を分けて整理する運用が適切です。

Q5

リリース後の保守費用と保守範囲を教えてください。

小規模〜中規模の社内利用を前提とした場合、 月額 3万円〜10万円程度を一つの目安と考えております。 実際の保守費用は、対応頻度、サポート体制、定例改善の有無により変動いたします。

  • 軽微な不具合修正
  • エラー発生時の原因調査
  • アカウント・権限管理の補助
  • ライブラリ更新・セキュリティアップデート対応
  • バックアップ状況や監視状況の確認

なお、新機能追加や大幅な仕様変更は通常、保守範囲には含めず別見積とするのが一般的です。

Q6

毎月のランニングコストが増加する条件を教えてください。

ランニングコストは、主に以下の条件で増加します。 60人規模では急激な増加は起こりにくいものの、機能拡張や保存データ量の増加に応じて段階的に上昇する想定です。

  • 利用ユーザー数の増加
  • プロフィール画像・添付ファイルなどの保存容量増加
  • 月間アクセス数・API呼び出し回数の増加
  • 通知メール送信数の増加
  • ログ保存期間やバックアップ保持期間の延長
  • 本番／検証／開発など複数環境の運用
  • 外部サービス連携（LINE、勤怠、会計ソフト等）の追加
  • 高度な監視サービスや分析基盤の導入
Q7

60人運用時の月額シミュレーション（年間運用・データ増加を加味した上限試算）をお願いします。

60名利用、プロフィール画像の保存、通知メールの運用、年間を通じたデータ蓄積を前提とした場合の概算は以下の通りです。

60名運用時の概算（月額）
Vercel Pro（ホスティング・CDN）約 ¥3,100 / 月
Supabase Pro（DB・認証・ストレージ）約 ¥3,900 / 月
メール送信（通知・パスワードリセット）約 ¥1,550 / 月
独自ドメイン按分約 ¥300 / 月
監視・ログ・予備費約 ¥2,000 〜 ¥5,000 / 月
月額合計目安約 ¥10,800 〜 ¥13,800 / 月
年間合計目安約 ¥129,600 〜 ¥165,600 / 年
上記は「実運用に耐える現実的な上限寄りの試算」です。利用頻度が穏やかな場合は、 これより低いコストで推移する可能性も十分ございます。

セキュリティに関するご質問

Q8

IP制限は可能ですか。

はい、可能です。 構成方法としては、Vercel 側またはリバースプロキシ／WAF 側での制御、 もしくはアプリケーション側での制御など、複数の方法があります。

例えば、「管理画面のみ社内IPからのアクセスに限定し、社員向け閲覧画面は外部からも利用可能」といった 切り分け運用も可能です。

ただし、外出先・スマートフォンからの利用を想定する場合は、IP制限を厳格にしすぎると利便性が下がるため、 対象画面・対象ユーザーごとに制御レベルを分ける方針が適切です。

Q9

不正アクセス検知はありますか。

標準機能のみで高度な SOC レベル監視を行うわけではありませんが、 ログイン失敗回数の記録、異常IPアクセスの監視、急激なアクセス増加の検知 など、 不正アクセス兆候を把握する仕組みは構築可能です。

  • 連続ログイン失敗の検知
  • 特定IPからの過剰アクセス検知
  • APIエラー急増の通知
  • Sentry / Logtail / Datadog 等との連携による監視強化
Q10

各ユーザーの操作ログは全て取得可能ですか。

はい、取得可能です。 ログイン、閲覧、作成、更新、削除、承認、通知送信等の各操作を 監査ログテーブルへ記録する設計にすることで、各ユーザーの操作履歴を追跡できます。

  • ユーザーID
  • 対象データID
  • 操作内容
  • 実行日時
  • IPアドレス
  • 端末・ブラウザ情報
  • 変更前後データ（必要に応じて）

なお、保存対象が増えるほど保管容量も増えるため、監査要件と個人情報保護の観点を踏まえ、 保存範囲を要件定義時に整理することが望ましいです。

Q11

バックアップの頻度・保存期間・復旧時間・復旧方法を教えてください。

想定運用としては、日次バックアップを基本とし、 保存期間は 7日〜30日程度を一つの基準と考えております。 要件に応じて、より長期の保存や別クラウドへの二重保存も対応可能です。

  • 頻度：日次（必要に応じて複数回/日も検討可）
  • 保存期間：7日〜30日を想定
  • 復旧時間目安：軽微障害は数十分〜数時間、重大障害は半日〜1営業日
  • 復旧方法：Supabase バックアップ復元、または SQL / CSV エクスポートから再投入

可用性を高める場合は、定期エクスポートを別ストレージへ保管する二重バックアップ体制も構築可能です。
