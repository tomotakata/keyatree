セキュリティ対応方針

現在の対応と今後の強化ポイント

SSL

HTTPS 常時暗号化

Vercel が自動で SSL 証明書を発行・更新。全通信が HTTPS で保護される。

ACL

権限ベースアクセス制御

5段階の権限（管理者〜閲覧専用）でページ・操作へのアクセスを制御。

AUTH

セッション管理

ログイン情報を Cookie（8時間有効）で管理。Middleware で未認証アクセスをブロック。

KEY

Supabase Auth（予定）

DB 連携時にパスワードを bcrypt でハッシュ化。メール認証・2段階認証を導入予定。

BKP

自動バックアップ（予定）

Supabase Pro プランで日次バックアップを自動取得。7日間のポイントインタイムリカバリ対応。

CD

CI/CD セキュリティ

GitHub の main ブランチ保護設定でレビューなしの直接プッシュを禁止（設定可能）。

追加質問への回答

Supabase・運用保守・ランニングコスト・セキュリティに関する補足回答

Supabase / Data Governance
更新履歴・削除履歴は追えるか
回答：標準DBだけでは「誰が・いつ・何を変更したか」を完全には残しませんが、監査ログ用テーブルを追加することで対応可能です。
具体的には、社員情報・評価・タスクなどの主要テーブルに対して、更新前データ / 更新後データ / 操作者 / 操作日時 / 操作種別（作成・更新・削除）を別テーブルへ自動記録します。
将来的に「履歴を元に戻す」「差分比較を表示する」運用も設計可能です。
監査ログ 変更履歴 削除履歴
Supabase / Performance
利用者 約60人で重くならないか
回答：60人規模の利用、および一定の同時アクセスが発生する前提でも、適切な設計を行えば十分に安定運用が可能です。
Vercel + Supabase は同時アクセスに対応しやすいクラウド構成であり、社員管理・評価管理・タスク管理といった業務用途であれば、60名規模は無理のないレンジです。
一覧画面のページネーション、画像最適化、集計処理のキャッシュ化、クエリ最適化を行うことで、同時利用時の負荷にも十分備えられます。
60名規模対応 同時アクセス対応 ページネーション キャッシュ
Supabase / Portability
将来 他クラウドへDB移行できるか
回答：可能です。Supabase は PostgreSQL ベースのため、将来的に AWS RDS / Google Cloud SQL / Azure Database for PostgreSQL / 自社サーバーPostgreSQL へ移行しやすい構成です。
ベンダーロックインを避けるため、アプリ側でも可能な限り標準SQL・標準PostgreSQL機能を中心に設計する方針が望ましいです。
画像ファイルは Storage、認証は Auth を使うため、その2点だけ移行時に追加対応が必要ですが、DB自体の移行性は高いです。
PostgreSQL互換 移行可能 ベンダーロック回避
Operations / SLA
障害発生時のサポート体制・対応時間
想定運用案：
・一次連絡手段：メール / Chatwork / Slack / 電話（緊急時）
・受付時間：平日 9:00〜18:00
・一次回答目安：重大障害 2時間以内、中障害 1営業日以内、軽微不具合 3営業日以内
・復旧対応：原因切り分け → 暫定対処 → 恒久対策 の順で対応
なお、正式なSLAは保守契約時に「障害レベル定義」と「対応時間帯」を明文化することを推奨します。
SLA案 平日9:00-18:00 重大障害2時間以内
Operations / Maintenance
保守費用と保守範囲の考え方
想定保守範囲：軽微修正、エラー調査、アカウント管理補助、ライブラリアップデート、バックアップ確認、監視確認。
概算：小規模運用では 月額 3万円〜10万円程度が目安です。
目標としては、初期フェーズでは「軽保守（障害時対応中心）」、本格運用後は「月次改善・定例対応込み」の運用保守へ切り替える形が現実的です。
大規模改修や新機能追加は保守費用に含めず、別見積とするのが一般的です。
軽保守 月3万〜10万円 新機能は別見積

ユーザー増 ストレージ増 通知増 環境増

Security / Access Control
IP制限は可能か
回答：可能です。方法は複数あり、用途に応じて選択します。
・Vercel / WAF / Reverse Proxy 側で許可IPのみアクセス許可
・ログイン後アプリ側でIPチェックして制御
・管理画面のみ社内IP制限、社員閲覧画面は外部許可という分離運用
ただし、外出先やスマホ利用を想定する場合は、IP制限を厳しくしすぎると利便性が下がるため、対象画面の切り分けが重要です。
IP制限可 管理画面のみ制限も可
Security / Detection
不正アクセス検知はあるか
回答：標準機能だけで完全なSOCレベル監視ではありませんが、ログイン失敗回数の記録、異常IPアクセス検出、急激なAPIリクエスト増加の監視、エラーログ通知などを組み合わせて検知可能です。
必要に応じて、Sentry / Logtail / Datadog 等の監視サービスを追加し、管理者に自動通知する構成に拡張できます。
異常検知 監視ツール連携可
Security / Audit
各ユーザーの操作ログは全て取得可能か
回答：可能です。ログイン、閲覧、作成、更新、削除、承認、通知送信などのイベントをすべて監査ログテーブルへ保存する設計にできます。
推奨項目は「ユーザーID」「対象データID」「操作内容」「日時」「IP」「端末情報」「変更前後データ」です。
実装時には、個人情報保護の観点から「何をどこまで保存するか」を事前に定義します。
操作ログ取得可 監査証跡
Backup / Recovery
バックアップ頻度・保存期間・復旧時間・復旧方法
想定方針：
・頻度：日次バックアップ（重要データは将来的に複数回/日も検討可）
・保存期間：7日〜30日（契約プランに応じて設定）
・復旧時間目安：軽微障害は数十分〜数時間、重大障害は半日〜1営業日
・復旧方法：Supabase のバックアップ復元、またはエクスポートSQL / CSV から再投入
より高い可用性が必要な場合は、定期エクスポートを別クラウドへ二重保存する運用も可能です。
日次バックアップ 7〜30日保存 二重保存可
